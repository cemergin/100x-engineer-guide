## The Goal

Payments are where engineering meets money, and money does not tolerate bugs. A bug in the event listing shows the wrong description for 5 minutes — annoying. A bug in payments double-charges 10,000 customers — lawsuit, chargeback fees, loss of payment processing privileges, and front-page news.

TicketPulse processes ticket purchases. Every purchase involves real money flowing from a customer's credit card through a payment provider (Stripe) to TicketPulse's merchant account. The invariants are absolute:

1. **No double charges.** If a customer clicks "Buy" twice, they are charged once.
2. **No lost money.** Every dollar that enters the system is accounted for.
3. **No phantom tickets.** A customer who was charged must have a ticket. A customer who was not charged must not.
4. **Full audit trail.** Every money movement is recorded and reconcilable.

This is a classic system design interview question, and for good reason — it touches idempotency, distributed transactions, ledger design, reconciliation, and failure handling. But here you are not answering an interview question. You are building it for TicketPulse.

**By the end of this module, you will have a payment system design that a fintech engineer would respect.**

### PCI Compliance: The Cardinal Rule

**TicketPulse must NEVER see raw card numbers.** Not in logs, not in the database, not in memory, not in transit.

Use Stripe Elements (or equivalent). The customer enters their card details into a Stripe-hosted iframe. Stripe returns a `payment_method_id` token. Your server only ever sees the token, never the card.

This keeps TicketPulse out of PCI DSS scope (mostly). Full PCI compliance for card-present data requires SAQ D certification — hundreds of security controls, quarterly scans, and annual audits. Using tokenized payment methods reduces this to SAQ A — a much simpler self-assessment.

## 1. Idempotency (20 minutes)

### The Problem

Networks are unreliable. A payment request can:
1. Succeed at Stripe but the response times out. The client retries. Without idempotency, the customer is charged twice.
2. Arrive twice due to a duplicate message in the queue. Without idempotency, the customer is charged twice.
3. Be retried by a load balancer after a failed health check. Without idempotency, the customer is charged twice.

Every payment system must be idempotent: processing the same request multiple times produces the same result as processing it once.

### The Idempotency Key

Every payment request includes a unique idempotency key generated by the client. If the server receives two requests with the same key, it returns the result of the first request without processing again.

```
Request 1: POST /api/payments { idempotency_key: "idem_abc123", amount: 5000 }
  → Process payment → Return { id: "pay_xyz", status: "succeeded" }

Request 2: POST /api/payments { idempotency_key: "idem_abc123", amount: 5000 }
  → Idempotency key found → Return { id: "pay_xyz", status: "succeeded" }
  → NO second charge
```

### 🛠️ Build: Implement Idempotency

```typescript
// src/payments/idempotency.ts

interface IdempotencyRecord {
  key: string;
  requestHash: string;      // Hash of the request body
  response: unknown;         // Cached response
  statusCode: number;
  createdAt: Date;
}

async function processPaymentIdempotently(
  idempotencyKey: string,
  requestBody: PaymentRequest,
  processPayment: () => Promise<PaymentResult>
): Promise<{ statusCode: number; body: unknown }> {

  const requestHash = crypto
    .createHash('sha256')
    .update(JSON.stringify(requestBody))
    .digest('hex');

  // Step 1: Check if we already processed this request
  const existing = await db.query(
    `SELECT response, status_code, request_hash
     FROM idempotency_keys
     WHERE key = $1`,
    [idempotencyKey]
  );

  if (existing.rows.length > 0) {
    const record = existing.rows[0];

    // Safety check: same key with different request body?
    if (record.request_hash !== requestHash) {
      return {
        statusCode: 422,
        body: {
          error: 'IDEMPOTENCY_KEY_REUSED',
          message: 'This idempotency key was used with a different request body',
        },
      };
    }

    // Return the cached response
    return {
      statusCode: record.status_code,
      body: record.response,
    };
  }

  // Step 2: Acquire a lock to prevent concurrent processing
  const lockAcquired = await redis.set(
    `lock:idem:${idempotencyKey}`,
    '1',
    'NX',  // Only set if not exists
    'EX',  // Expire
    30     // 30 second lock timeout
  );

  if (!lockAcquired) {
    return {
      statusCode: 409,
      body: {
        error: 'REQUEST_IN_PROGRESS',
        message: 'A request with this idempotency key is already being processed',
      },
    };
  }

  try {
    // Step 3: Process the payment in a single transaction
    const result = await db.transaction(async (tx) => {
      // Insert the idempotency record (will fail on duplicate key)
      await tx.query(
        `INSERT INTO idempotency_keys (key, request_hash, created_at, expires_at)
         VALUES ($1, $2, NOW(), NOW() + INTERVAL '24 hours')
         ON CONFLICT (key) DO NOTHING`,
        [idempotencyKey, requestHash]
      );

      // Process the actual payment
      const paymentResult = await processPayment();

      // Store the response for future idempotent lookups
      await tx.query(
        `UPDATE idempotency_keys
         SET response = $1, status_code = $2
         WHERE key = $3`,
        [JSON.stringify(paymentResult), 201, idempotencyKey]
      );

      return paymentResult;
    });

    return { statusCode: 201, body: result };
  } finally {
    await redis.del(`lock:idem:${idempotencyKey}`);
  }
}
```

### The Database Schema for Idempotency

```sql
CREATE TABLE idempotency_keys (
    key           VARCHAR(255) PRIMARY KEY,
    request_hash  VARCHAR(64) NOT NULL,
    response      JSONB,
    status_code   INT,
    created_at    TIMESTAMP NOT NULL DEFAULT NOW(),
    expires_at    TIMESTAMP NOT NULL
);

-- Clean up expired keys daily
CREATE INDEX idx_idempotency_expires ON idempotency_keys (expires_at);

-- TTL: 24 hours. After that, the same idempotency key can be reused.
-- This matches Stripe's idempotency key retention period.
```

### 💡 Insight: Stripe's Idempotency at Scale

Stripe processes over $1 trillion annually. Their idempotency key system handles approximately 1% of all requests being retries. That is billions of idempotent replay requests per year. The system works because:
- Every API endpoint accepts an `Idempotency-Key` header.
- Keys are retained for 24 hours.
- The response is cached and replayed exactly — including errors.
- Even failed requests are cached. If the first attempt returned a validation error, the retry returns the same error (not a duplicate charge).

## 2. The Ledger (20 minutes)

### Double-Entry Bookkeeping

Every money movement in TicketPulse creates exactly two ledger entries: a debit and a credit. The sum of all entries must always be zero. This is how banks, payment processors, and every serious financial system works.

**Why double-entry?**
- Single-entry: "Customer paid $50." Where did the $50 go? No record.
- Double-entry: "Customer account debited $50. Merchant account credited $50." The money is tracked from source to destination. The books always balance.

### 🛠️ Build: Implement a Simple Ledger

```sql
-- The ledger table (append-only, immutable)
CREATE TABLE ledger_entries (
    id                BIGSERIAL PRIMARY KEY,
    transaction_id    UUID NOT NULL,          -- groups related entries
    payment_intent_id UUID NOT NULL,
    account_id        UUID NOT NULL,          -- which account
    account_type      VARCHAR(30) NOT NULL,   -- customer, merchant, platform, reserve
    entry_type        VARCHAR(10) NOT NULL,   -- DEBIT or CREDIT
    amount            BIGINT NOT NULL,        -- always positive, in cents
    currency          VARCHAR(3) NOT NULL,
    description       VARCHAR(255),
    created_at        TIMESTAMP NOT NULL DEFAULT NOW()
);

-- CRITICAL: This table is append-only. No UPDATEs. No DELETEs.
-- To reverse a transaction, INSERT new entries (a reversal).

-- Index for reconciliation queries
CREATE INDEX idx_ledger_payment ON ledger_entries (payment_intent_id);
CREATE INDEX idx_ledger_account ON ledger_entries (account_id, created_at);
CREATE INDEX idx_ledger_transaction ON ledger_entries (transaction_id);
```

### Recording a Ticket Purchase

```typescript
async function recordPurchaseInLedger(
  tx: DatabaseTransaction,
  paymentIntentId: string,
  customerId: string,
  merchantId: string,
  amountInCents: number,
  currency: string
): Promise<void> {
  const transactionId = uuidv4();
  const platformFee = Math.round(amountInCents * 0.05); // 5% platform fee
  const merchantAmount = amountInCents - platformFee;

  // Entry 1: Debit customer (money leaves their card)
  await tx.query(
    `INSERT INTO ledger_entries
     (transaction_id, payment_intent_id, account_id, account_type, entry_type, amount, currency, description)
     VALUES ($1, $2, $3, 'customer', 'DEBIT', $4, $5, 'Ticket purchase')`,
    [transactionId, paymentIntentId, customerId, amountInCents, currency]
  );

  // Entry 2: Credit merchant (money goes to event organizer)
  await tx.query(
    `INSERT INTO ledger_entries
     (transaction_id, payment_intent_id, account_id, account_type, entry_type, amount, currency, description)
     VALUES ($1, $2, $3, 'merchant', 'CREDIT', $4, $5, 'Ticket sale revenue')`,
    [transactionId, paymentIntentId, merchantId, merchantAmount, currency]
  );

  // Entry 3: Credit platform (TicketPulse's fee)
  await tx.query(
    `INSERT INTO ledger_entries
     (transaction_id, payment_intent_id, account_id, account_type, entry_type, amount, currency, description)
     VALUES ($1, $2, $3, 'platform', 'CREDIT', $4, $5, 'Platform fee')`,
    [transactionId, paymentIntentId, 'ticketpulse-platform', platformFee, currency]
  );

  // Verify: debits must equal credits
  // DEBIT: amountInCents
  // CREDITS: merchantAmount + platformFee = (amountInCents - platformFee) + platformFee = amountInCents
  // ✓ Balanced
}
```

### The Invariant Check

Run this every hour. If it ever returns a non-zero result, something is critically wrong:

```sql
-- The Golden Query: must ALWAYS return 0
SELECT
  SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE 0 END) -
  SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END) AS balance
FROM ledger_entries;

-- If this returns anything other than 0, stop everything and investigate.
```

For per-transaction verification:

```sql
-- Every transaction must balance independently
SELECT
  transaction_id,
  SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE 0 END) AS total_debits,
  SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END) AS total_credits
FROM ledger_entries
GROUP BY transaction_id
HAVING SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE 0 END) !=
       SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END);

-- This query should ALWAYS return 0 rows.
```

## 3. Reconciliation (10 minutes)

### What Is Reconciliation?

Reconciliation is the process of comparing TicketPulse's ledger with the payment provider's records. You have two sources of truth: your database and Stripe's database. They should agree. When they do not, you have a problem that must be resolved.

### 📐 Design: The Reconciliation Process

```
Daily at 2:00 AM UTC:

1. EXPORT from TicketPulse:
   SELECT payment_intent_id, amount, currency, status, created_at
   FROM payments
   WHERE created_at BETWEEN '2026-03-23 00:00:00' AND '2026-03-24 00:00:00'

2. EXPORT from Stripe:
   stripe.paymentIntents.list({ created: { gte: 1742688000, lt: 1742774400 } })

3. COMPARE:
   For each Stripe transaction:
     - Does it exist in TicketPulse's ledger?
     - Do the amounts match?
     - Do the statuses match?

4. FLAG mismatches:
   Category A: In TicketPulse but not in Stripe
     → Our system recorded a payment that Stripe does not know about.
     → Likely: payment was recorded optimistically before Stripe confirmation.
     → Action: Check Stripe for the payment_intent. If not found, reverse the ledger entry.

   Category B: In Stripe but not in TicketPulse
     → Stripe charged the customer but our system does not know about it.
     → THIS IS THE WORST CASE. Customer was charged, has no ticket.
     → Action: IMMEDIATE alert to ops team. Create the ticket manually. Investigate root cause.

   Category C: Amount mismatch
     → Stripe charged a different amount than our ledger shows.
     → Likely: currency conversion issue or race condition.
     → Action: Investigate and create an adjustment entry.

5. RESOLVE:
   Auto-resolve known patterns (Category A with pending status).
   Escalate unknowns to the operations team within 1 hour.
```

### 📐 Design Exercise: What Happens When They Don't Match?

For each mismatch category, design the automated resolution and the escalation path. Think about:
- What does the customer experience?
- What does the operations team need to do?
- What is the SLA for resolution?

## 4. The Refund Flow (10 minutes)

### Refunds Are Not Simply "Reversing a Payment"

A refund involves:
1. **Reverse the ledger entries** (new entries, not deleting old ones — the ledger is append-only).
2. **Request refund from Stripe** (Stripe processes the refund to the customer's card).
3. **Handle the delay** (Stripe refunds take 5-10 business days to appear on the customer's statement).
4. **Update the ticket status** (ticket becomes available again if the event has not passed).

```typescript
async function processRefund(
  orderId: string,
  reason: string
): Promise<RefundResult> {
  return db.transaction(async (tx) => {
    // 1. Find the original payment
    const order = await tx.query(
      'SELECT * FROM orders WHERE id = $1 FOR UPDATE',
      [orderId]
    );

    if (!order.rows[0]) throw new Error('Order not found');
    if (order.rows[0].status === 'refunded') {
      // Idempotent: already refunded
      return { status: 'already_refunded', orderId };
    }

    const payment = await tx.query(
      'SELECT * FROM payments WHERE order_id = $1',
      [orderId]
    );

    // 2. Reverse the ledger entries
    const transactionId = uuidv4();
    const amount = payment.rows[0].amount;
    const merchantId = order.rows[0].merchant_id;
    const customerId = order.rows[0].user_id;
    const platformFee = Math.round(amount * 0.05);
    const merchantAmount = amount - platformFee;

    // Credit customer (money returns to their card)
    await tx.query(
      `INSERT INTO ledger_entries
       (transaction_id, payment_intent_id, account_id, account_type, entry_type, amount, currency, description)
       VALUES ($1, $2, $3, 'customer', 'CREDIT', $4, 'usd', $5)`,
      [transactionId, payment.rows[0].id, customerId, amount, `Refund: ${reason}`]
    );

    // Debit merchant (money taken back from event organizer)
    await tx.query(
      `INSERT INTO ledger_entries
       (transaction_id, payment_intent_id, account_id, account_type, entry_type, amount, currency, description)
       VALUES ($1, $2, $3, 'merchant', 'DEBIT', $4, 'usd', $5)`,
      [transactionId, payment.rows[0].id, merchantId, merchantAmount, `Refund: ${reason}`]
    );

    // Debit platform (return the fee)
    await tx.query(
      `INSERT INTO ledger_entries
       (transaction_id, payment_intent_id, account_id, account_type, entry_type, amount, currency, description)
       VALUES ($1, $2, $3, 'platform', 'DEBIT', $4, 'usd', $5)`,
      [transactionId, payment.rows[0].id, 'ticketpulse-platform', platformFee, `Refund: ${reason}`]
    );

    // 3. Request refund from Stripe
    const stripeRefund = await stripe.refunds.create({
      payment_intent: payment.rows[0].stripe_payment_intent_id,
      reason: 'requested_by_customer',
    });

    // 4. Update order and ticket status
    await tx.query(
      'UPDATE orders SET status = $1, refunded_at = NOW() WHERE id = $2',
      ['refunded', orderId]
    );

    await tx.query(
      'UPDATE tickets SET status = $1, reserved_by = NULL, purchased_at = NULL WHERE id = $2',
      ['available', order.rows[0].ticket_id]
    );

    return {
      status: 'refunded',
      orderId,
      stripeRefundId: stripeRefund.id,
      amount,
    };
  });
}
```

### What If the Stripe Refund Fails?

The ledger entries are already written. The Stripe refund call fails (network timeout, Stripe outage).

**Do not roll back the ledger entries.** Instead:
1. Record the refund as `pending_provider` in your system.
2. A background job retries the Stripe refund with exponential backoff.
3. When Stripe confirms the refund (via webhook), update the status to `refunded`.
4. If retries exhaust (e.g., card expired, account closed), escalate to operations for manual resolution.

## 5. The Worst-Case Scenario (10 minutes)

### 🐛 Debug: Payment Succeeds at Stripe, Callback Times Out

This is the nightmare scenario every payment engineer fears:

```
1. Customer clicks "Buy"
2. TicketPulse calls Stripe → Stripe charges the card → $50 deducted
3. Stripe returns success response to TicketPulse
4. TicketPulse's server crashes / network timeout BEFORE recording the payment
5. Customer sees an error page
6. Customer was charged $50 but has no ticket
```

### How to Handle This

**Defense 1: Idempotency Key**

The client generated an idempotency key. The customer retries (or clicks "Buy" again). The retry hits TicketPulse with the same idempotency key. TicketPulse calls Stripe with the same key. Stripe recognizes the key and returns the original successful charge without charging again. TicketPulse records the payment and issues the ticket.

**Defense 2: Stripe Webhooks**

Even if the customer never retries, Stripe sends a `payment_intent.succeeded` webhook to TicketPulse's webhook endpoint. The webhook handler:
1. Looks up the payment intent in TicketPulse's database.
2. If not found: creates the payment record, issues the ticket, sends confirmation email.
3. If found: verifies status matches. No action needed.

```typescript
// Webhook handler
app.post('/webhooks/stripe', async (req, res) => {
  const event = stripe.webhooks.constructEvent(
    req.body,
    req.headers['stripe-signature'],
    WEBHOOK_SECRET
  );

  if (event.type === 'payment_intent.succeeded') {
    const paymentIntent = event.data.object;

    // Check if we already processed this
    const existing = await db.query(
      'SELECT * FROM payments WHERE stripe_payment_intent_id = $1',
      [paymentIntent.id]
    );

    if (existing.rows.length === 0) {
      // We missed this payment! Record it now.
      console.error(
        `[RECONCILIATION] Payment ${paymentIntent.id} was charged but not recorded. Processing now.`
      );
      await recordPaymentAndIssueTicket(paymentIntent);
    }
  }

  res.json({ received: true });
});
```

**Defense 3: Reconciliation**

The daily reconciliation job catches anything that both idempotency and webhooks missed. Category B mismatches (in Stripe but not in TicketPulse) trigger an immediate alert.

### The Defense-in-Depth Stack

```
Defense 1: Idempotency key    → Handles retries (most common case)
Defense 2: Stripe webhooks    → Handles crashes/timeouts (async recovery)
Defense 3: Daily reconciliation → Catches anything that slipped through
Defense 4: Ops alert + manual → Last resort for truly anomalous cases
```

No single defense is sufficient. Together, they ensure that no payment is ever lost.

## 6. Reflect (10 minutes)

### 🤔 Questions

1. **What is more important for a payment system: availability or consistency? Why?**

   Think carefully. If the payment system is unavailable, customers cannot buy tickets — lost revenue. If the payment system is inconsistent, customers are double-charged or undercharged — lawsuits, chargebacks, regulatory penalties.

   **The answer:** Consistency. A payment system that is unavailable for 5 minutes during a Stripe outage loses some sales but causes no lasting damage. A payment system that double-charges 10,000 customers causes lasting damage to trust, brand, and the company's relationship with payment processors. This is why payment systems are CP (consistent + partition-tolerant), not AP.

2. **Why is the ledger append-only? Why not just UPDATE the balance when a payment is made?**

   Because updates destroy history. With append-only entries, you have a complete audit trail. You can reconstruct the state of any account at any point in time. Regulators require this. Fraud detection depends on it. Debugging requires it.

3. **A high-profile event sells 50,000 tickets in 30 seconds. Each ticket costs $200. That is $10M in 30 seconds. What could go wrong with the payment system under this load?**

   Think about: database connection pool exhaustion, Stripe rate limits (25 requests/second per API key in test mode, higher in production with agreement), idempotency key storage growing rapidly, ledger write throughput, and the risk of partial failures during the surge.

4. **TicketPulse wants to expand to Japan. Japanese customers prefer Konbini (convenience store) payments, which take 24-48 hours to complete. How does this change the payment flow?**

   Think about: ticket reservation duration, ledger entries for pending payments, what happens if the customer never pays, and how this interacts with the saga pattern.

## Glossary

| Term | Definition |
|------|-----------|
| **Idempotency key** | A unique identifier for a payment request that ensures the same request processed multiple times produces the same result. |
| **Double-entry bookkeeping** | An accounting method where every financial transaction creates two entries (a debit and a credit) that must sum to zero. |
| **Ledger** | An immutable, append-only record of all financial transactions. The source of truth for money movements. |
| **Reconciliation** | The process of comparing internal records with external records (e.g., Stripe) to find and resolve discrepancies. |
| **PCI DSS** | Payment Card Industry Data Security Standard. The security standard for organizations that handle credit card information. |
| **Payment intent** | A Stripe concept representing the lifecycle of a payment: created → confirmed → succeeded/failed. |
| **Chargeback** | When a customer disputes a charge with their bank. The bank reverses the charge and the merchant loses the money plus a fee ($15-100). |
| **SAQ** | Self-Assessment Questionnaire. A PCI compliance validation tool. SAQ A (simplest) applies when you use hosted payment fields. |
